Local AI for automation

automation / ai inference

My interest in local AI for automation came from wanting models to run on my own machine instead of depending on an online API. One clear example was a computer-vision automation project built around an ONNX YOLOv8 model.

The goal was not just to write a simple script that clicked fixed coordinates. I wanted a system that could visually understand what was happening on screen: UI states, icons, confirmation prompts, menu elements, and interaction targets.

The first challenge was defining what the model actually needed to detect. For a human, recognizing an icon or a button is instant. For a model, every object has to become a class with consistent labels and enough examples.

I had to think about screenshots, bounding boxes, class definitions, confidence thresholds, false positives, false negatives, and how visual elements can change depending on resolution, scaling, animations, effects, or background clutter.

OpenCV handled the frame-processing side: capturing the screen, preparing images, filtering regions, and passing the right input into the model. ONNX made the inference portable and local, which meant the project did not need to send screenshots or visual data to a cloud service.

Running everything locally also made the limitations obvious. Local inference has to respect hardware constraints: GPU memory, model size, inference speed, accuracy, and stability. A model that works well in a notebook is not automatically useful in a real-time loop.

This changed how I think about AI automation. The model matters, but the surrounding software matters just as much: preprocessing, timing, decision logic, fallback behavior, logging, and debugging tools. Running AI locally is also about owning the pipeline and understanding its constraints.
